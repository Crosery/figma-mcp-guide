17 Tools + 7 Skills 完全使用指南

Official Remote MCP · mcp.figma.com · Plugin: figma@claude-plugins-official · 2026.04

架构：MCP Tools vs Skills

Figma MCP 由两层组成：底层的 17 个原子 API 工具 + 上层的 7 个工作流 Skill。日常用 Skill（套餐），精细控制用 Tool（单点）。

底层 MCP Tools — 17 个原子 API

  • 由 mcp.figma.com 远程服务器提供
  • 每个工具做一件事：读截图、取元数据、写画布...
  • 需要手动传参（fileKey, nodeId 等）
  • 需要自己编排调用顺序
  • 类比：单个 API endpoint

上层 Skills — 7 个工作流模板

  • 由 figma@claude-plugins-official 插件提供
  • 封装完整工作流 + 最佳实践 + 错误处理
  • 自动编排多个 Tool 的调用顺序
  • 通过 /skill-name 触发
  • 类比：封装好的 SDK 方法
原则：能用 Skill 就用 Skill，Skill 覆盖不了的场景再直接调 Tool。

从 Figma 读取设计数据 — 6 个工具

T01 get_design_context

mcp__plugin_figma_figma__get_design_context

核心工具 — 设计转代码的主力。给一个 Figma 节点，返回 React+Tailwind 参考代码 + 截图 + Code Connect 提示 + 资源下载 URL。这是你最常用的工具。

关键参数

参数说明
nodeId *节点 ID，如 "123:456"（URL 中的 - 转 :）
fileKey *从 URL /design/ 后提取
excludeScreenshot省略截图节省 context token
disableCodeConnect不走 Code Connect 映射
forceCode节点太大时强制返回代码
clientFrameworks"react", "vue" 等（用于日志）

使用技巧

  • 返回的是参考代码，需适配你的技术栈（Vue/React/Swift...）
  • 节点太大只返回 metadata，加 forceCode=true 强制
  • 默认包含截图，除非设 excludeScreenshot
  • 如果有 Code Connect 映射，会直接返回你的组件代码
  • URL 解析：branch URL 用 branchKey 作 fileKey
  • 返回的资源 URL 有 7 天有效期，及时下载
// 从 URL 提取参数
// https://figma.com/design/ABC123/MyApp?node-id=12-34
// → fileKey = "ABC123", nodeId = "12:34"

// 返回内容包含：
// 1. code — React+Tailwind 参考代码
// 2. screenshot — 节点截图
// 3. assets — { imgLogo: "https://figma.com/api/mcp/asset/..." }
// 4. hints — Code Connect / 组件文档链接 / 设计注释

T02 get_screenshot

mcp__plugin_figma_figma__get_screenshot

生成指定节点的截图。轻量级预览工具 — 只看设计不生成代码时用它。

参数

参数说明
nodeId *节点 ID
fileKey *文件 Key

使用技巧

  • 比 get_design_context 轻量很多，只返回图片
  • 适合实现前确认"选对节点没"
  • 实现后对比验证"代码和设计一不一致"
  • 配合 get_metadata：先看结构 → 截图确认 → 取代码
  • 大页面建议分段截图（按 section nodeId），全页截图分辨率低会漏细节
get_design_context 默认就含截图。单独用 get_screenshot 的场景：只看不写代码、验证实现结果、逐段检查大页面。

T03 get_metadata

mcp__plugin_figma_figma__get_metadata

获取节点的 XML 结构元数据 — 节点 ID、图层类型、名称、位置、尺寸。没有样式细节，但结构一目了然。适合大文件先看骨架。

参数

参数说明
nodeId *可传页面 ID "0:1" 看整页
fileKey *文件 Key

使用技巧

  • 传 nodeId="0:1" 获取整个页面结构概览
  • 从结果中找到目标节点 ID 再调 get_design_context
  • 不要对 Figma Make 文件使用
  • get_design_context 返回截断时，先用它找子节点 ID
  • 返回 XML 格式，包含层级关系

T04 get_figjam

mcp__plugin_figma_figma__get_figjam

专门读取 FigJam 白板（figma.com/board/...）的内容。可读取便签、连接线、形状等元素，将白板内容转为结构化数据。

参数

参数说明
nodeId *"0:1" 获取全部
fileKey *文件 Key
includeImagesOfNodes包含节点图片 (默认 true)

使用技巧

  • 只适用于 FigJam 文件（URL 含 /board/）
  • 普通 Design 文件不能用此工具
  • 适合将头脑风暴/流程图转为代码逻辑
  • 设 includeImagesOfNodes=false 减少响应体积

T05 get_variable_defs

mcp__plugin_figma_figma__get_variable_defs

获取节点关联的 Figma Variable 定义 — 颜色、字体、间距等设计 Token。返回格式如 {'color/primary': '#a259ff', 'spacing/lg': '24px'}。

参数

参数说明
nodeId *节点 ID
fileKey *文件 Key

使用技巧

  • 提取设计系统 Token 值，映射到 CSS 变量 / Tailwind
  • 确保代码和设计用同一套颜色/间距
  • 配合 search_design_system 找完整 Token 体系
  • 返回的是 Variable 定义，不只是当前节点用了哪些

T06 search_design_system

mcp__plugin_figma_figma__search_design_system

跨所有已链接库搜索设计系统资产 — 组件、变量、样式。用关键词找 Button、color、spacing 等现成资源。

参数

参数说明
query *搜索关键词
fileKey *文件 Key
includeComponents含组件 (默认 true)
includeVariables含变量 (默认 true)
includeStyles含样式 (默认 true)
includeLibraryKeys限定搜索范围到特定库

使用技巧

  • 创建组件前先搜索！避免重复造轮子
  • 搜索跨所有已订阅库，不只是当前文件
  • 多用短关键词："button" "gray" "space" "shadow"
  • 返回 component key 可用 importComponentByKeyAsync 导入
  • 搜颜色时设 includeComponents=false 减少噪音
  • 用 includeLibraryKeys 过滤只看特定库

T07 use_figma

mcp__plugin_figma_figma__use_figma

万能写入工具 — 在 Figma 文件中执行 Plugin API JavaScript。可创建/编辑/删除任何 Figma 对象。50KB 代码限制。

参数

参数说明
fileKey *目标文件 Key
code *Plugin API JS 代码
description *代码功能描述

关键规则 (必记)

  • 必须先加载 /figma-use skill 再调用！
  • 用 return 返回数据，不用 figma.closePlugin()
  • 颜色 0-1 范围，不是 0-255
  • Inter "Semi Bold" 有空格！不是 "SemiBold"
  • 换页用 await figma.setCurrentPageAsync()
  • FILL 必须在 appendChild 之后设置
  • 文字操作前必须 loadFontAsync
  • 失败时 STOP，不要盲目重试
use_figma 是默认写入工具。唯一例外：首次从网页捕获用 generate_figma_design。更新已捕获页面仍用 use_figma。

CODE CONNECT

Figma 组件与代码组件的双向映射 — 5 个工具

T11-15 Code Connect 工具组

5 个工具协同完成 Figma 组件 ↔ 代码组件的映射。映射后 get_design_context 返回你的真实组件代码而非通用代码。

5 个工具各司其职

工具作用
get_code_connect_suggestionsAI 自动建议哪些组件该映射
get_context_for_code_connect获取组件属性/Variant/子节点树
add_code_connect_map添加单个映射（支持 16 种框架）
send_code_connect_mappings批量保存多个映射
get_code_connect_map查询已有映射关系

标准工作流

  • Step 1: get_code_connect_suggestions → 找出未映射组件
  • Step 2: get_context_for_code_connect → 看组件有哪些 props
  • Step 3: 在代码库中找对应组件
  • Step 4: add_code_connect_map 或 send_code_connect_mappings 保存
  • Step 5: get_code_connect_map 验证映射成功
支持框架：React, Vue, Swift, SwiftUI, Compose, Kotlin, Flutter, Web Components, Svelte, Storybook, JS, Java, PHP, Markdown 等 16 种
映射完成后，get_design_context 会自动返回你项目中的 <Button variant="primary"> 而非通用 Tailwind 代码。一次投入，长期收益。

7 个上层工作流 — 封装 Tools 的最佳实践

S01 /figma-implement-design

Figma 设计 → 项目代码 (1:1 复刻)

最常用的 Skill — 给一个 Figma 链接，按 7 步标准流程生成生产级代码。自动适配你的技术栈、复用项目组件、下载资源、验证视觉一致性。

7 步标准流程

  • Step 1: 解析 URL → 提取 fileKey + nodeId
  • Step 2: get_design_context → 获取代码+截图+Token
  • Step 3: get_screenshot → 视觉参考基准
  • Step 4: 下载资源（图片/SVG/图标）
  • Step 5: 翻译为项目技术栈（Vue/React/...）
  • Step 6: 追求 1:1 视觉还原
  • Step 7: 对照截图验证 Checklist

使用技巧

  • 节点太大被截断 → 自动 get_metadata 找子节点分批取
  • 优先复用项目已有组件，不重复造轮子
  • Figma Token 映射到项目 Token，不硬编码
  • 资源 URL 来自 Figma 资源端点，直接下载
  • 偏差记录在代码注释中说明原因
// 使用方式
/figma-implement-design
"帮我用 Vue3 实现这个设计：https://figma.com/design/xxx/yyy?node-id=12-34"

S02 /figma-use

use_figma 的强制前置 Skill

每次调用 use_figma 工具前必须先加载此 Skill。它提供 Plugin API 的完整规则、坑点、参考文档索引。不加载直接调会出各种难以调试的错误。

它加载了什么

  • 14 条关键规则（return 模式、颜色范围、页面切换...）
  • 页面上下文重置规则（每次调用重置到第一页）
  • 错误恢复协议（失败 = 原子性，无残留）
  • 增量工作流指导（一步一验证）
  • 飞行前检查清单（提交前 14 项必查）
  • 10 份参考文档的索引（按需加载）

核心要点速查

  • 用 return 返回数据 + 所有创建的 node ID
  • 不能用 figma.notify()、console.log()
  • 颜色 0-1，不是 0-255
  • await 每个 Promise，不能 fire-and-forget
  • FILL/HUG 必须在 appendChild 之后
  • 字体用前必须 loadFontAsync
  • 失败 = 未执行，安全重试
不要跳过这个 Skill！调 use_figma 前必须 /figma-use。Skill 里有 plugin-api-standalone.d.ts 可以 grep 查任何 API 签名。

S03 /figma-generate-design

代码/描述 → Figma 设计画布

将代码页面或文字描述转化为 Figma 设计稿。核心理念：不画裸矩形，而是发现并复用设计系统中的组件、变量、样式，逐段组装画面。

6 步工作流

  • Step 1: 理解画面 — 读源码，列出各段落和组件
  • Step 2: 发现设计系统 — search_design_system 找组件/变量/样式
  • Step 3: 创建 Page Wrapper Frame
  • Step 4: 逐段构建（每段一个 use_figma 调用）
  • Step 5: 截图验证全页
  • Step 6: 更新已有画面（定位→修改→验证）

关键原则

  • 绝不硬编码 hex 颜色 / px 间距 — 用变量绑定
  • 优先 importComponentByKeyAsync 复用组件实例
  • 用 setBoundVariable / setBoundVariableForPaint 绑 Token
  • 每段用 get_screenshot 验证，不要全做完才看
  • 文字覆盖用 setProperties() 而非直接改 characters
  • Web 应用可并行 generate_figma_design 截取做参考

S04 /figma-generate-library

从代码库构建完整的 Figma 设计系统

大型工程 Skill — 需要 20~100+ 次 use_figma 调用，分 5 个阶段构建专业级设计系统。包含 Token 架构、组件变体、文档页面。每个阶段有用户确认检查点。

5 阶段流程

  • Phase 0: Discovery — 分析代码 + 检查 Figma + 锁定范围
  • Phase 1: Foundations — 创建 Variable 集合/模式/别名/样式
  • Phase 2: File Structure — 页面骨架 + 基础文档
  • Phase 3: Components — 逐个组件（原子→分子顺序）
  • Phase 4: Integration QA — Code Connect + 审计

关键规则

  • 永远不要一步到位 — 最小单元 + 验证
  • Variable 必须在 Component 之前创建
  • 每个 Variable 必须设 scopes（不用 ALL_SCOPES）
  • 每个 Variable 设 code syntax (var(--xxx))
  • 语义层 alias 到 Primitive 层，不复制值
  • Variant 矩阵 > 30 就拆分子组件
  • 状态持久化到 /tmp/dsb-state-{RUN_ID}.json
  • 每阶段必须用户确认才能继续

S05 /figma-code-connect

创建 .figma.js Code Connect 模板文件

将 Figma 组件映射到代码库中的真实组件。创建 parserless 模板文件（.figma.js），让 Figma Dev Mode 直接显示项目组件代码。需要 Organization/Enterprise 计划。

6 步流程

  • Step 1: 解析 Figma URL
  • Step 2: get_code_connect_suggestions 发现未映射组件
  • Step 3: get_context_for_code_connect 获取属性定义
  • Step 4: 在代码库中找到匹配的组件
  • Step 5: 创建 .figma.js 模板（属性映射）
  • Step 6: 验证模板覆盖所有属性

属性映射方法

Figma 属性模板方法
TEXTinstance.getString('Name')
BOOLEANinstance.getBoolean('Name', {true:..., false:...})
VARIANTinstance.getEnum('Name', {'FigmaVal': 'codeVal'})
INSTANCE_SWAPinstance.getInstanceSwap('Name')
子图层instance.findInstance('LayerName')
文本图层instance.findText('Name').textContent
必须是已发布到 Team Library 的组件才能用 Code Connect。Free/Professional 计划不支持。

S06 /figma-create-design-system-rules

生成项目专属的设计系统规则

分析你的代码库结构、组件模式、样式方案，生成一份规则文件保存到 CLAUDE.md / AGENTS.md / .cursor/rules/。后续所有 Figma 转代码都自动遵循项目规范。

5 步流程

  • Step 1: 调用 create_design_system_rules 获取模板
  • Step 2: 分析代码库 — 组件位置/命名/样式/Token
  • Step 3: 生成项目专属规则
  • Step 4: 保存到对应的规则文件
  • Step 5: 用简单组件测试验证

规则覆盖范围

  • 组件规则: 位置、命名、导出模式
  • 样式规则: CSS 方案、Token 位置、禁硬编码
  • Figma 流程: 必须先 get_design_context 再实现
  • 资源规则: localhost 资源直接用、不装新图标库
  • 项目规范: 路由/状态管理/import 约定
项目初始化时跑一次就够了。相当于给 AI 写了一份"项目新人入职手册"。

S07 /figma-to-localui

Figma 节点样式 → Vue 组件 CSS 变量同步

你自己定制的 Skill — 从 Figma 选区实时读取样式值（尺寸/圆角/颜色/字体），和 Vue 组件的 CSS 变量对比，精准更新 variables.css。支持本地插件和官方远程双 MCP。

5 步工作流

  • Step 1: Figma 选区 → get_selection + get_node_info
  • Step 2: 提取样式值（尺寸/圆角/填充/描边/文字）
  • Step 3: 在代码中找对应 Vue 组件和 CSS 变量
  • Step 4: 对比差异 → 修改 variables.css
  • Step 5: 资源替换（SVG 图标/背景图）

双 MCP 策略

场景用哪个 MCP
日常样式对比本地插件（无限次/实时）
下载资源图片官方远程（7 天 URL）

Vue 项目实战工作流

从零开始用 Figma MCP 复刻设计的完整路径。

一次性设置（做一次就够）

  • 1. 生成规则: /figma-create-design-system-rules
    → 分析项目结构，生成 CLAUDE.md 规范
  • 2. 提取 Token: get_variable_defs
    → 映射到 CSS 变量 / Tailwind / UnoCSS
  • 3. 组件映射: /figma-code-connect
    → Button.figma.js, Input.figma.js ...

日常开发循环

  • 1. 实现设计: /figma-implement-design [URL]
    → 自动 7 步流程，生成 Vue SFC
  • 2. 样式微调: /figma-to-localui
    → 选中 Figma 节点，同步 CSS 变量差异
  • 3. 代码 → Figma: /figma-generate-design
    → 实现后同步回 Figma 保持双向一致
